# Banksy Database Schema Reference

PostgreSQL 15+. All primary keys are UUIDs generated by the database
(`gen_random_uuid()`). All timestamps are stored without timezone.

The schema is managed by Flyway migrations in
`src/main/resources/db/migration/`. The canonical source of truth is always
those files — this document reflects their current cumulative state across all
four migrations (V1–V4).

---

## Entity Relationship Overview

```
users ──────────────────────────────────────────────────┐
  │                                                      │
  │ 1:many                                               │ 1:many (owner)
  ▼                                                      ▼
oauth_identities            user_plaid_items ──────► plaid_items
                            (junction)                   │
                                                         │ 1:many
                                                         ▼
                                                    plaid_accounts

users ──► notifications
  1:many
```

- A `user` can link many bank connections (`plaid_items`) via the
  `user_plaid_items` junction table.
- A `plaid_item` has exactly one `owner_user_id` but can appear in
  `user_plaid_items` for multiple users (shared bank connections).
- A `plaid_item` contains one or more `plaid_accounts` (e.g. a Chase
  connection might have both a checking and savings account).
- A `user` can have many `notifications`.
- A `user` can have many `oauth_identities` (one per OAuth provider).

## Tables

---

### `users`

Core identity table. No passwords — authentication is handled entirely by
OAuth. A user row is created on first login.

| Column | Type | Nullable | Default | Notes |
|--------|------|----------|---------|-------|
| `id` | `UUID` | NO | `gen_random_uuid()` | Primary key |
| `first_name` | `VARCHAR(100)` | NO | — | |
| `last_name` | `VARCHAR(100)` | NO | — | |
| `username` | `VARCHAR(50)` | NO | — | Unique |
| `email` | `VARCHAR(255)` | NO | — | Unique |
| `created_at` | `TIMESTAMP` | NO | `NOW()` | Set once on insert |
| `updated_at` | `TIMESTAMP` | NO | `NOW()` | Updated on every change |

**API exposure:** `id`, `firstName`, `lastName`, `username`, `email` are
returned by `GET /api/auth/me`.

**TypeScript**
```
User {
  id:         string    // UUID
  firstName:  string
  lastName:   string
  username:   string
  email:      string
  createdAt:  string    // ISO timestamp — internal, not in any API response
  updatedAt:  string    // ISO timestamp — internal, not in any API response
}
```

### `oauth_identities`

Links a `users` row to a specific OAuth provider identity. One user can have
multiple identities if they log in with different providers (currently only
Google is configured).

| Column | Type | Nullable | Default | Notes |
|--------|------|----------|---------|-------|
| `id` | `UUID` | NO | `gen_random_uuid()` | Primary key |
| `user_id` | `UUID` | NO | — | FK → `users.id` ON DELETE CASCADE |
| `provider` | `VARCHAR(50)` | NO | — | e.g. `"google"` |
| `provider_user_id` | `VARCHAR(255)` | NO | — | The user's ID within the provider (e.g. Google sub) |
| `created_at` | `TIMESTAMP` | NO | `NOW()` | |

**Constraints:** `UNIQUE (provider, provider_user_id)` — prevents the same
provider account from being linked to two different users.

**Indexes:** `idx_oauth_identities_user_id`

**API exposure:** Never exposed. Internal auth plumbing only.

**TypeScript**
```
OAuthIdentity {
  id:             string   // UUID — internal only
  userId:         string   // UUID
  provider:       string   // e.g. "google"
  providerUserId: string   // internal only
  createdAt:      string   // ISO timestamp — internal only
}
```

### `plaid_items`

One row represents one connected bank institution (what Plaid calls an "Item").
A single item can be shared across multiple users via `user_plaid_items`.

| Column | Type | Nullable | Default | Notes |
|--------|------|----------|---------|-------|
| `id` | `UUID` | NO | `gen_random_uuid()` | Primary key — **our internal ID** |
| `access_token_enc` | `TEXT` | NO | — | AES-256-GCM encrypted Plaid access token. **Never expose to the frontend.** |
| `item_id` | `VARCHAR(255)` | NO | — | Plaid's own item identifier. Unique. Used for deduplication on re-link. |
| `institution_id` | `VARCHAR(255)` | NO | — | Plaid's institution identifier (e.g. `"ins_3"` for Chase) |
| `institution_name` | `VARCHAR(255)` | NO | — | Human-readable bank name (e.g. `"Chase"`) |
| `transaction_cursor` | `TEXT` | YES | `NULL` | Plaid pagination cursor for incremental transaction sync. Currently unused — reserved for future use. |
| `owner_user_id` | `UUID` | NO | — | FK → `users.id`. The user who originally linked this bank. Only the owner can relink or share. |
| `status` | `VARCHAR(20)` | NO | `'HEALTHY'` | Health state — see enum below |
| `created_at` | `TIMESTAMP` | NO | `NOW()` | |
| `updated_at` | `TIMESTAMP` | NO | `NOW()` | |

**`status` enum values:**

| Value | Meaning | Frontend action |
|-------|---------|----------------|
| `HEALTHY` | Token is valid, data can be fetched | None needed |
| `NEEDS_REAUTH` | User's bank session expired (Plaid `ITEM_LOGIN_REQUIRED`) | Show relink prompt — use update-mode link token |
| `INVALID_TOKEN` | Token was fully revoked (Plaid `INVALID_ACCESS_TOKEN`) | Show relink prompt — use full-relink token, or offer removal |

**Indexes:** `idx_plaid_items_owner_user_id`

**API exposure:** `id` (as `plaidItemId` in `RelinkSignal`), `institutionName`
(in `RelinkSignal`), `status` (drives `errorType` in `RelinkSignal`).
`access_token_enc`, `item_id`, `institution_id`, `transaction_cursor` are never
sent to the frontend.

**TypeScript**
```
PlaidItemStatus = "HEALTHY" | "NEEDS_REAUTH" | "INVALID_TOKEN"

PlaidItem {
  id:               string           // UUID — our internal ID; used in API paths
  itemId:           string           // Plaid's ID — internal only, never in API responses
  institutionId:    string           // internal only
  institutionName:  string           // exposed in RelinkSignal
  status:           PlaidItemStatus  // drives RelinkSignal.errorType
  ownerUserId:      string           // UUID — internal only
  createdAt:        string           // ISO timestamp — internal only
  updatedAt:        string           // ISO timestamp — internal only
  // accessTokenEnc and transactionCursor are never exposed
}
```

### `plaid_accounts`

One row represents one individual bank account within a `plaid_item` (e.g.
a Chase checking account and a Chase savings account are two rows, both
belonging to the same Chase `plaid_item`).

| Column | Type | Nullable | Default | Notes |
|--------|------|----------|---------|-------|
| `id` | `UUID` | NO | `gen_random_uuid()` | Primary key — **our internal ID** |
| `plaid_item_id` | `UUID` | NO | — | FK → `plaid_items.id` ON DELETE CASCADE |
| `plaid_account_id` | `VARCHAR(255)` | NO | — | Plaid's own account identifier. Used internally to filter hidden accounts from Plaid API responses. Unique. |
| `name` | `VARCHAR(255)` | NO | — | Short account name (e.g. `"Plaid Checking"`) |
| `official_name` | `VARCHAR(255)` | YES | `NULL` | Full official account name from the bank, if provided |
| `type` | `VARCHAR(50)` | NO | — | Account category (e.g. `"depository"`, `"credit"`, `"investment"`) |
| `subtype` | `VARCHAR(50)` | YES | `NULL` | Account sub-category (e.g. `"checking"`, `"savings"`, `"credit card"`) |
| `mask` | `VARCHAR(4)` | YES | `NULL` | Last 4 digits of the account number. Safe to store and display. |
| `hidden` | `BOOLEAN` | NO | `FALSE` | Soft-hide flag. Hidden accounts are excluded from balance and transaction responses. |
| `created_at` | `TIMESTAMP` | NO | `NOW()` | |

**Indexes:** `idx_plaid_accounts_item_id`

**API exposure:** `id` (as `plaidAccountId` path param for `PUT /api/plaid/account/:id/hide`),
`mask` (included in hide-success notification message — not directly in any
API response to the frontend today). `plaid_account_id` is internal only.

**TypeScript**
```
PlaidAccount {
  id:             string          // UUID — our internal ID; used as plaidAccountId in hide endpoint
  plaidItemId:    string          // UUID
  plaidAccountId: string          // Plaid's ID — internal only
  name:           string
  officialName:   string | null
  type:           string          // "depository" | "credit" | "investment" | ...
  subtype:        string | null   // "checking" | "savings" | "credit card" | ...
  mask:           string | null   // last 4 digits, e.g. "4242"
  hidden:         boolean
  createdAt:      string          // ISO timestamp — internal only
}
```

### `user_plaid_items`

Junction table that grants a user access to a bank connection. This is what
makes bank sharing possible — if two users share a bank, both have a row here
pointing to the same `plaid_item`. There is only ever one access token per bank,
stored on the `plaid_item` itself.

| Column | Type | Nullable | Default | Notes |
|--------|------|----------|---------|-------|
| `user_id` | `UUID` | NO | — | FK → `users.id` ON DELETE CASCADE |
| `plaid_item_id` | `UUID` | NO | — | FK → `plaid_items.id` ON DELETE CASCADE |
| `added_at` | `TIMESTAMP` | NO | `NOW()` | When this user gained access to this bank |

**Primary key:** `(user_id, plaid_item_id)` composite

**Indexes:** `idx_user_plaid_items_user_id`, `idx_user_plaid_items_plaid_item_id`

**API exposure:** Never exposed directly. The existence of a row here is what
determines whether a user sees a bank in their balance/transaction responses.

**TypeScript**
```
UserPlaidItem {
  userId:       string   // UUID — internal only
  plaidItemId:  string   // UUID — internal only
  addedAt:      string   // ISO timestamp — internal only
}
```

### `notifications`

In-app notifications created by the backend after remove-bank operations
(both successful removals and failures). One notification is created per
affected user — if a bank is shared by three users and gets deleted, all
three receive a notification.

| Column | Type | Nullable | Default | Notes |
|--------|------|----------|---------|-------|
| `id` | `UUID` | NO | `gen_random_uuid()` | Primary key |
| `user_id` | `UUID` | NO | — | FK → `users.id` ON DELETE CASCADE |
| `message` | `TEXT` | NO | — | Human-readable message, safe to display |
| `created_at` | `TIMESTAMP` | NO | `NOW()` | Set automatically on insert |
| `read` | `BOOLEAN` | NO | `FALSE` | Whether the user has read this notification |

**Indexes:** `idx_notifications_user_id`

**API exposure:** No notification endpoint exists yet. This table is populated
but not queryable from the frontend. When a notifications endpoint is built,
these are the fields it will expose.

**TypeScript**
```
Notification {
  id:        string    // UUID
  userId:    string    // UUID
  message:   string
  createdAt: string    // ISO timestamp
  read:      boolean
}
```

## Complete Schema (current state after all migrations)

```sql
-- V1 — initial schema
CREATE TABLE users (
    id          UUID          PRIMARY KEY DEFAULT gen_random_uuid(),
    first_name  VARCHAR(100)  NOT NULL,
    last_name   VARCHAR(100)  NOT NULL,
    username    VARCHAR(50)   NOT NULL UNIQUE,
    email       VARCHAR(255)  NOT NULL UNIQUE,
    created_at  TIMESTAMP     NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMP     NOT NULL DEFAULT NOW()
);

CREATE TABLE oauth_identities (
    id                UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id           UUID         NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    provider          VARCHAR(50)  NOT NULL,
    provider_user_id  VARCHAR(255) NOT NULL,
    created_at        TIMESTAMP    NOT NULL DEFAULT NOW(),
    UNIQUE (provider, provider_user_id)
);
CREATE INDEX idx_oauth_identities_user_id ON oauth_identities(user_id);

CREATE TABLE plaid_items (
    id                  UUID          PRIMARY KEY DEFAULT gen_random_uuid(),
    access_token_enc    TEXT          NOT NULL,
    item_id             VARCHAR(255)  NOT NULL UNIQUE,
    institution_id      VARCHAR(255)  NOT NULL,
    institution_name    VARCHAR(255)  NOT NULL,
    transaction_cursor  TEXT,
    created_at          TIMESTAMP     NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMP     NOT NULL DEFAULT NOW()
);

CREATE TABLE plaid_accounts (
    id                UUID          PRIMARY KEY DEFAULT gen_random_uuid(),
    plaid_item_id     UUID          NOT NULL REFERENCES plaid_items(id) ON DELETE CASCADE,
    plaid_account_id  VARCHAR(255)  NOT NULL UNIQUE,
    name              VARCHAR(255)  NOT NULL,
    official_name     VARCHAR(255),
    type              VARCHAR(50)   NOT NULL,
    subtype           VARCHAR(50),
    mask              VARCHAR(4),
    created_at        TIMESTAMP     NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_plaid_accounts_item_id ON plaid_accounts(plaid_item_id);

CREATE TABLE user_plaid_items (
    user_id        UUID       NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    plaid_item_id  UUID       NOT NULL REFERENCES plaid_items(id) ON DELETE CASCADE,
    added_at       TIMESTAMP  NOT NULL DEFAULT NOW(),
    PRIMARY KEY (user_id, plaid_item_id)
);
CREATE INDEX idx_user_plaid_items_user_id       ON user_plaid_items(user_id);
CREATE INDEX idx_user_plaid_items_plaid_item_id ON user_plaid_items(plaid_item_id);

-- V2 — owner and status on plaid_items
ALTER TABLE plaid_items
    ADD COLUMN owner_user_id UUID        NOT NULL REFERENCES users(id),
    ADD COLUMN status        VARCHAR(20) NOT NULL DEFAULT 'HEALTHY';
CREATE INDEX idx_plaid_items_owner_user_id ON plaid_items(owner_user_id);

-- V3 — notifications
CREATE TABLE notifications (
    id          UUID      PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID      NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    message     TEXT      NOT NULL,
    created_at  TIMESTAMP NOT NULL DEFAULT NOW(),
    read        BOOLEAN   NOT NULL DEFAULT FALSE
);
CREATE INDEX idx_notifications_user_id ON notifications(user_id);

-- V4 — hidden flag on plaid_accounts
ALTER TABLE plaid_accounts ADD COLUMN hidden BOOLEAN NOT NULL DEFAULT FALSE;
```

## ID Naming — Internal vs. Plaid's Own IDs

Two tables have both an internal UUID and a Plaid-assigned string ID. These are
different and must not be confused:

| Table | Our internal ID | Plaid's ID | Which one the API uses |
|-------|----------------|------------|----------------------|
| `plaid_items` | `id` (UUID) | `item_id` (VARCHAR) | Our `id` — used as `plaidItemId` in `RelinkSignal` and the delete endpoint |
| `plaid_accounts` | `id` (UUID) | `plaid_account_id` (VARCHAR) | Our `id` — used as `plaidAccountId` in the hide endpoint |

The Plaid-assigned IDs (`item_id`, `plaid_account_id`) are internal only and
are never sent to the frontend.

## What the Frontend Never Sees

These columns exist in the database but are never included in any API response:

| Column | Why |
|--------|-----|
| `plaid_items.access_token_enc` | Encrypted secret — must never leave the server |
| `plaid_items.item_id` | Plaid's internal identifier — not needed by the frontend |
| `plaid_items.institution_id` | Plaid's institution identifier — not needed by the frontend |
| `plaid_items.transaction_cursor` | Internal pagination state for future sync feature |
| `plaid_accounts.plaid_account_id` | Plaid's internal identifier — used server-side only for filtering |
| `oauth_identities.*` | Entire table is auth plumbing — internal only |
| `user_plaid_items.*` | Entire table is access control — internal only |
